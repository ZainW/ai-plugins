# Autospec Plugin Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the `autospec` Claude Code plugin — takes freeform specs, reviews them for gaps, asks clarifying questions, routes to implementation based on prescriptiveness, and delivers verified code.

**Architecture:** Separate plugin in the `ai-plugins` monorepo alongside `autoresearch`. Same patterns: SKILL.md orchestrator, markdown agents with YAML frontmatter, JSONL state logging, session-resume hook. Five agents (analyzer, interrogator, researcher, implementer, verifier) orchestrated by a single `/spec` skill.

**Tech Stack:** Claude Code plugin system (markdown skills + agents), Node.js (hooks), git worktrees (isolation)

**Spec:** `docs/superpowers/specs/2026-03-22-autospec-design.md`

### Task 2: Analyzer Agent

**Files:**
- Create: `autospec/agents/analyzer/analyzer.md`
- Reference: `autoresearch/agents/researcher/researcher.md` (for frontmatter format)

The analyzer is the first agent in the pipeline. It receives raw spec text + codebase context and returns two things: a prescriptiveness classification (prescriptive/directional/open-ended) and a categorized gap analysis.

- [ ] **Step 1: Write the analyzer agent**

Create `autospec/agents/analyzer/analyzer.md` with YAML frontmatter matching the pattern used in autoresearch agents:

```yaml
---
name: analyzer
description: >
  Review a freeform spec for completeness and classify its prescriptiveness.
  Detects gaps across four categories (technical, criteria, context, scope),
  tags each as blocking or nice-to-clarify, and classifies the spec as
  prescriptive, directional, or open-ended.
model: sonnet
tools: Read, Grep, Glob, WebSearch
maxTurns: 20
---
```

Body should cover:
- **Your Job**: Analyze the raw spec and codebase context. Produce classification + gap analysis.
- **Prescriptiveness Classification**: Define the three levels with examples (from spec: prescriptive = names specific technologies + detailed acceptance criteria, directional = describes what but leaves technical choices open, open-ended = describes problem/goal without approach).
- **Gap Detection**: Four categories (technical, criteria, context, scope). Each gap gets a severity: `blocking` or `nice-to-clarify`.
- **Codebase Analysis**: Instruct the agent to use Grep/Glob to find relevant existing code, check `package.json` for dependencies, understand what's already built. Use WebSearch to check if named technologies are current/deprecated.
- **Output Format**: Structured text (not file writes) with exact format:

```
## SPEC ANALYSIS

### Classification: [prescriptive|directional|open-ended]
**Confidence:** [high|medium|low]
**Reasoning:** [2-3 sentences why]

### Gap Analysis

#### Blocking Gaps (must resolve before proceeding)
1. [TECHNICAL|CRITERIA|CONTEXT|SCOPE] — [description]
2. ...

#### Nice-to-Clarify (can assume defaults)
1. [TECHNICAL|CRITERIA|CONTEXT|SCOPE] — [description] — **Default assumption:** [what we'd assume]
2. ...

### Codebase Context
- **Relevant files:** [list]
- **Current dependencies:** [relevant ones from package.json]
- **Patterns detected:** [frameworks, conventions, existing solutions]
- **Conflicts/concerns:** [anything in the spec that clashes with current codebase]

### Technology Checks
- [tech mentioned in spec]: [current/deprecated/unknown] — [notes]
```

- **Important Rules**: Do NOT write to any files. Return analysis as text output. Be thorough with gap detection — the interrogator depends on complete analysis. If WebSearch is unavailable, note "unable to verify" for technology checks rather than guessing.

Expected: Same YAML structure (name, description, model, tools, maxTurns)

### Task 3: Interrogator Agent

**Files:**
- Create: `autospec/agents/interrogator/interrogator.md`

The interrogator takes the analyzer's gap analysis and converts it into natural, conversational questions for the user.

- [ ] **Step 1: Write the interrogator agent**

Create `autospec/agents/interrogator/interrogator.md` with frontmatter:

```yaml
---
name: interrogator
description: >
  Convert gap analysis into natural, conversational questions for the user.
  Groups related gaps, proposes smart defaults, references codebase context.
  Feels like a sharp tech lead reviewing your spec, not a robotic checklist.
model: sonnet
tools: Read, Grep, Glob
maxTurns: 15
---
```

Body should cover:
- **Your Job**: You receive the analyzer's gap analysis (and on subsequent rounds, the user's previous answers). Generate targeted questions to fill the gaps.
- **Input**: On first round: analyzer output + raw spec. On subsequent rounds: also includes previous Q&A pairs and remaining blocking gaps.
- **Tone**: Sharp tech lead, not robotic. Conversational, direct, opinionated about defaults.
- **Grouping Rules**: Combine related gaps into single questions. Don't ask 15 separate questions when 5 grouped ones cover it.
- **Smart Defaults**: For `nice-to-clarify` gaps, propose a default and let user override. Format: "I'll assume X unless you say otherwise."
- **Codebase References**: Use Read/Grep/Glob to reference what you find. "You're already using Prisma — should I add fields to the existing schema?"
- **Output Format**:

```
## QUESTIONS (Round N)

### Blocking (must answer)
1. [Question referencing the gap] — Context: [why this matters]
2. ...

### Defaults (will assume unless you disagree)
- [assumption 1]
- [assumption 2]
- ...

### Gaps Resolved This Round: [N]
### Blocking Gaps Remaining: [N]
```

- **Important Rules**: Do NOT write to any files. Keep questions concise. Maximum 7 questions per round. If there are no blocking gaps remaining, say so explicitly.


### Task 4: Researcher Agent

**Files:**
- Create: `autospec/agents/researcher/researcher.md`
- Reference: `autoresearch/agents/researcher/researcher.md` (for comparison — autospec's is simpler)

Autospec's researcher is lighter than autoresearch's. It answers focused technical questions rather than doing full candidate evaluations with weighted scoring.

- [ ] **Step 1: Write the researcher agent**

Create `autospec/agents/researcher/researcher.md` with frontmatter:

```yaml
---
name: researcher
description: >
  Focused research on a specific technical question arising from a spec.
  Answers questions like "What's the best WebSocket library for this stack?"
  without full candidate evaluation. Returns findings with a recommendation.
model: sonnet
tools: WebSearch, WebFetch, Read, Grep, Glob, Bash
maxTurns: 20
---
```

Body should cover:
- **Your Job**: Research a specific technical question and return a recommendation. Unlike autoresearch's researcher, you're not doing a full candidate evaluation with weighted scores — you're answering a focused question.
- **Input**: A specific technical question, codebase context (frameworks, dependencies, patterns), and the relevant section of the refined spec.
- **Process**:
  1. Understand the question and codebase constraints
  2. WebSearch for current best practices, libraries, patterns
  3. Check codebase for compatibility (existing deps, framework, TypeScript, etc.)
  4. Evaluate 2-3 options briefly
  5. Make a clear recommendation
- **Output Format**:

```
## RESEARCH FINDINGS: [topic]

### Recommendation: [name/approach]
**Why:** [2-3 sentences]

### Options Considered
1. **[option]** — [one-line assessment]
2. **[option]** — [one-line assessment]
3. **[option]** — [one-line assessment]

### Key Facts
- [relevant stat or fact]
- [compatibility note]
- [any gotchas]

### Impact on Spec
[How this recommendation affects the implementation — what to add to spec.md]
```

- **Important Rules**: Do NOT write to any files. Use REAL data from web searches. Note "unable to verify" rather than fabricating. Be opinionated — give a clear recommendation, not a wishy-washy comparison.


```yaml
name: implementer
```

Body should follow the same structure as `autoresearch/agents/implementer/implementer.md` but with these key differences:
- **Input description**: Receives refined `spec.md` + implementation plan (not research findings/evaluation)

- **Process**: Same incremental commit approach as autoresearch's implementer
- **Auto-fix retry prompt template**: Same as autoresearch but references spec.md instead of research
- **Spec fidelity rule**: "If the spec says X, implement X. If you think X is wrong, implement X anyway and note your concern in the implementation summary. The spec was reviewed and approved — don't second-guess it during implementation."
- **Output Format**: Same `IMPLEMENTATION SUMMARY` format as autoresearch's implementer
- **Important Rules**: Same rules (commit frequently, don't break incrementally, follow code style, don't over-engineer)

### Task 8: Orchestrator Skill (SKILL.md)

**Files:**
- Create: `autospec/skills/spec/SKILL.md`
- Reference: `autoresearch/skills/research/SKILL.md` (same orchestrator pattern)
- Reference: `docs/superpowers/specs/2026-03-22-autospec-design.md` (the spec — this IS the source of truth)

This is the most complex file — it's the orchestrator that drives the entire pipeline. Write it section by section following the spec exactly.

- [ ] **Step 1: Write the skill frontmatter + header**

```yaml
---
name: spec
description: >
  Spec-driven coding for Claude Code. Paste a spec from any source (Google Docs,
  Jira, Slack, markdown, email), get reviewed and verified code. Reviews specs for
  gaps, asks clarifying questions, routes to implementation based on prescriptiveness.
  Trigger phrases: "spec", "implement this spec", "here's the spec", "my boss sent",
  "build from spec", "execute this spec".
argument-hint: "<paste or describe the spec>"
user-invocable: true
---
```

- [ ] **Step 2: Write Quick Reference + JSONL Event Schema sections**

Same pattern as autoresearch's SKILL.md. Include:
- State directory paths (`.autospec/`)
- Session doc, structured log, config, spec.md paths
- Branch naming convention (`autospec/<task-slug>-<date>`)
- Full JSONL event schema from the design spec (all event types with their fields)

- [ ] **Step 3: Write Resume Check section**

Follow the resume logic table from the design spec exactly. Same pattern as autoresearch — read `.autospec/session.md`, check `.autospec/log.jsonl` for last phase, skip to appropriate phase. Include all event types from the spec's resume table.

Include `/spec clear` cleanup: delete `.autospec/`, remove `autospec/*` branches, remove autospec worktrees.

- [ ] **Step 4: Write Phase 1 (Intake + Analysis)**

1. Parse the task from `$ARGUMENTS`
2. **Minimum input threshold**: If raw spec is under ~50 words with no concrete requirements, prompt user for more detail before proceeding
3. Scan codebase (Grep/Glob for relevant files, check `package.json`)
4. Read config from `.autospec/config.json` if it exists
5. Dispatch the `analyzer` agent with the raw spec + codebase context
6. Collect analyzer output (classification + gap analysis)
7. Initialize state: create `.autospec/`, add to `.gitignore`, write initial `session.md`, append `intake` event to `log.jsonl`

Note: Unlike autoresearch Phase 1, there is NO user checkpoint here — the analysis results flow directly into Phase 2 where the user interacts.

- [ ] **Step 5: Write Phase 2 (Interrogation)**

Interrogation loop mechanism (from spec):
1. Dispatch interrogator agent with analyzer output
2. Present questions to user, collect answers
3. Check: blocking gaps remaining?
   - Yes + rounds < `maxInterrogationRounds` (default 5): re-dispatch interrogator with original analysis + all answers so far
   - Yes + rounds >= max: proceed with best-effort defaults, warn user
   - No: proceed to spec refinement
4. Log each round as `interrogation` event
5. After all gaps filled: write refined spec to `.autospec/spec.md`, append `spec_refined` event, update `session.md`

Include the `session.md` template (adapted from autoresearch):
```markdown
# Autospec: <task summary>

## Objective
<What we're building and why>

## Classification
<prescriptive/directional/open-ended>

## Gap Analysis Summary
<Key gaps found and how they were resolved>

## Refined Spec
See .autospec/spec.md

## Research Findings
<To be updated if research is needed>

## Implementation Plan
<To be updated after Phase 4>

## Implementation Log
<To be updated after Phase 5>

## Verification Results
<To be updated after Phase 6>
```

- [ ] **Step 6: Write Phase 3 (Routing Decision)**

Present routing to user based on classification:

For **prescriptive**: Inform user spec is clear, going straight to implementation plan. No choices needed, but user can request research if they want.

For **directional**: Present 3 options (quick research recommended, skip research, full research). Dispatch autospec's own `researcher` agent for light research. Fold results into `spec.md`.

For **open-ended**: Present 3 options (full research recommended, light research, skip research).

For full research: check if `../autoresearch/agents/researcher/researcher.md` exists.
- If yes: read autoresearch agent prompts, dispatch parallel researchers + evaluator following autoresearch's Phase 2-3 pattern, present options to user, fold selected approach into `spec.md`
- If no: fall back to autospec's researcher for light research, warn user

Log `routing` event. If research happens, also log `research_routed` event (for full research path), then `research` events per researcher, and `evaluation` event after evaluator runs.

- [ ] **Step 7: Write Phase 4 (Implementation Plan)**

Present a concrete plan to the user:
- Files to create and modify
- Order of operations
- Key decisions derived from the spec
- Assumptions being made

Wait for user approval. Log `plan` event.

- [ ] **Step 8: Write Phase 5 (Implementation)**

Build implementer prompt with:
- Refined `spec.md` content
- Implementation plan details
- Files in scope
- Codebase context
- Constraints

Include auto-fix retry prompt template (for Phase 6 loop):
- Verification failure report
- Focus on fixing flagged issues only
- Reference spec.md for context

Dispatch implementer agent. Collect summary. Append `implement` event. Update `session.md`.

- [ ] **Step 9: Write Phase 6 (Verification + Auto-fix + Final Report)**

Same pattern as autoresearch Phases 6-8:
1. Dispatch verifier agent
2. Evaluate results: PASS → final report, FAIL → auto-fix, WARN → final report with warnings
3. Auto-fix loop: append `autofix` event, re-dispatch implementer with fix prompt, re-verify, max retries from config
4. If max retries exceeded: `complete` with `status: "escalated"`, present failure options
5. On success: `complete` with `status: "success"`, present final report

Final report template from the design spec (Autospec Complete with spec summary, classification, branch, verification results, changes, next steps).

- [ ] **Step 10: Write Rules section**

Same rules as autoresearch's SKILL.md:
- Only YOU write to state files
- Dispatch researchers in PARALLEL
- User checkpoints: interrogation (Phase 2) and implementation plan (Phase 4)
- Log everything to `log.jsonl`
- On resume, read `log.jsonl` and pick up from last event
- `/spec clear` removes all state

- [ ] **Step 11: Verify the complete SKILL.md**

Read through the entire file. Check:
- All phases from the spec are covered
- JSONL event types match the schema section
- Resume logic matches the resume table
- Agent names match the agent files in `autospec/agents/`
- No references to autoresearch-specific concepts that don't apply
